# Agentic RAG System: Smart Documentation Assistant

## 📝 Project Reflection

* **Challenges:** One of the primary challenges was ensuring the **Router** accurately distinguishes between semantic technical queries and structured data requests. Initially, the system lacked temporal awareness, making it difficult to answer "what is the latest rule" type of questions.
* **Key Improvements:** By expanding the **Pydantic Schema** to include `source_file`, `observed_at`, and `line_range`, and upgrading the router to act as a **Structured Data Engine**, the system's reliability and precision significantly increased.
* **Workflow Success:** The implementation of an **Event-Driven Workflow** with a built-in **Retry Mechanism** proved to be highly effective. It allows the agent to self-correct when retrieval confidence is low, ensuring a higher quality of response.
* **Future Work:** * Implementing automated **Unit Tests** (using Pytest) to validate the router's classification logic.
    * Adding a real-time monitor to trigger the **Offline Extraction** process automatically whenever a documentation file is updated.

## 📌 Project Overview
This project implements an **Event-Driven Agentic RAG (Retrieval-Augmented Generation) System**. It is designed to automatically ingest, process, and answer complex queries based on technical documentation and markdown files generated by various coding agents (such as Kiro and Windsurf). 

Unlike standard RAG systems that rely solely on semantic similarity, this Agentic RAG introduces a **Smart Routing Architecture** capable of classifying user intent and selecting the optimal retrieval strategy, including querying a Vector Database or retrieving strictly formatted offline JSON data.

## 🛠️ Tech Stack
* **Framework:** LlamaIndex (Core, Workflows, Agents)
* **LLM & Embeddings:** Cohere (`command-a-03-2025` and `embed-multilingual-v3.0`)
* **Vector Database:** Pinecone
* **User Interface:** Gradio
* **Data Structuring:** Pydantic

## 🧠 Smart Routing Architecture
The core intelligence of this system lies in its `agent_logic` router, which classifies queries into three distinct execution paths:

1. **STRUCTURED PATH (Offline Data Extraction):**
   * **Use Case:** When a user asks for comprehensive lists (e.g., "List all rules" or "Show all warnings").
   * **Mechanism:** Bypasses semantic search completely. Reads from a pre-extracted `extracted_data.json` file generated offline by the LLM using Pydantic schemas (`RuleItem`, `WarningItem`, `DecisionItem`). Ensures 100% coverage without being limited by vector chunk sizes.
   
2. **SEMANTIC PATH (Vector Search):**
   * **Use Case:** When a user asks specific, technical questions requiring deep context (e.g., "How is the database configured?").
   * **Mechanism:** Triggers an Event-Driven `RAGWorkflow`. Retrieves the top-K chunks from Pinecone. Includes self-correction logic: if the retrieval confidence score is below a certain threshold, the system automatically rewrites the query and retries the search.

3. **GENERAL PATH (Standard LLM):**
   * **Use Case:** Greetings, general knowledge, or casual conversation.
   * **Mechanism:** Bypasses project documentation entirely and responds using the base LLM, saving retrieval latency and tokens.

## 🚀 Setup & Installation

1. **Clone the repository:**
   ```bash
   git clone <your-repo-url>
   cd <your-project-directory>
   ```

2. **Install requirements:**
   ```bash
   pip install llama-index llama-index-llms-cohere llama-index-embeddings-cohere llama-index-vector-stores-pinecone pinecone-client gradio python-dotenv pydantic
   ```

3. **Environment Variables:**
   Create a `.env` file in the root directory and add your API keys:
   ```env
   COHERE_API_KEY=your_cohere_key_here
   PINECONE_API_KEY=your_pinecone_key_here
   ```

## 💻 How to Run

Open the `main.ipynb` Jupyter Notebook and run the cells in the following logical order:
1. **Data Ingestion:** Run the cells that load the markdown files, generate embeddings, and upsert vectors to Pinecone. *(Ensure metadata like the `tool` source is added).*
2. **Offline Extraction:** Run the `extract_and_save_offline_data()` cell **once** to parse all documents and generate the `extracted_data.json` file.
3. **Launch UI:** Run the final cell containing the `gr.ChatInterface` to launch the Gradio web interface.
